<objective>
Close OPT-04: enable the WHOOP 5 (Gen5) / MG optical sensor stream on Android so optical
frames (packet_k 20/21/26) actually arrive and reach the Rust bridge, achieving parity
with the iOS v15.0 optical decode path.

Research (HIGH confidence) confirmed the routing path is already correct — Gen5/MG
handleNotification forwards ALL frames to importFrame → GooseBridge.safeHandle() with no
packet_k filtering (D-01). The single real gap is the upstream command layer: Android never
sends ENABLE_OPTICAL_DATA (cmd 107) or TOGGLE_OPTICAL_MODE (cmd 108), so the device never
starts emitting optical packets. iOS sends both together in startPhysiologyCapture.

Purpose: Without these two BLE commands the WHOOP 5 stays silent on optical channels and
the otherwise-correct Android routing has nothing to route. This delivers the Android side
of optical capture parity.

Output: Two optical enable commands added to the Gen5/MG handshake (Gen4 excluded), plus a
JVM unit test that pins both the optical command wire format and the no-filter routing
contract.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Android app → WHOOP device (BLE command characteristic) | App writes optical enable commands to an already-paired GATT peripheral |
| WHOOP device → Android (BLE notify characteristic) → Rust bridge | Device-emitted optical frame bytes cross into the app and then into Rust for parsing |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-117-01 | Tampering | Inbound optical frame bytes from device | accept | Frame content is validated/parsed by the Rust core (GooseBridge.safeHandle); Android performs no trust-bearing logic on the bytes. No new validation needed on Android. |
| T-117-02 | Information Disclosure | Optical enable BLE writes | accept | Write-only commands ([0x01,0x01]) to an already-paired device; no secrets or PII in the payload. |
| T-117-03 | Denial of Service | Gen4 receiving Gen5-only optical commands | mitigate | Generation guard (Gen5/MG only) prevents sending cmd 107/108 to Gen4, avoiding GATT write errors / undefined device behavior (research Pitfall 3). |
| T-117-04 | Denial of Service | Command collision with historical sync sequence | mitigate | Dedicated sensor sequence counter (start 180) separate from syncSequence + 500ms delay so historical sync queues first (research Pitfall 2). |
| T-117-SC | Tampering | npm/pip/cargo installs | mitigate | No package installs in this phase; uses existing buildCommandFrame, WhoopGeneration enum, and JUnit 4 already present. No legitimacy gate required. |
</threat_model>
