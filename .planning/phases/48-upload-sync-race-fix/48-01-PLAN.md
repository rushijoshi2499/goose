    Test structure (directive — do not copy literally, write idiomatic Rust inline):

    1. Call `make_store()` to get an in-memory GooseStore.
    2. Execute a raw SQL INSERT into `hr_samples` with device_id='dev-race', ts=1.0, bpm=70 — this is the "pre-upload" row.
    3. Call `store.rows_pending_upload("hr_samples", 500)` → collect the returned Vec, extract rowid values into a Vec<i64> named `captured_ids`. Assert `captured_ids.len() == 1`.
    4. Execute a second raw SQL INSERT into `hr_samples` with device_id='dev-race', ts=2.0, bpm=72 — this is the "race-window" row that arrives AFTER pre-capture but BEFORE mark.
    5. Call `store.mark_synced_rows("hr_samples", &captured_ids)`. Assert return value is Ok(1).
    6. Call `store.rows_pending_upload("hr_samples", 10)` again → the result must have exactly 1 row. Assert `pending.len() == 1`.
    7. Extract `ts` from `pending[0]` as f64. Assert it equals 2.0 (the race-window row).
    8. Verify the pre-captured row is now synced=1 by executing `SELECT synced FROM hr_samples WHERE ts=1.0` and asserting the value is 1i64.

    The test uses `store.conn.execute(...)` for raw inserts (same pattern as `test_mark_synced_sets_flag` at line 9176).
    The `rows_pending_upload` return type is `GooseResult<Vec<serde_json::Value>>` — extract rowid via `.as_i64()`.
    Annotate with `#[test]` and a doc comment explaining the race-window scenario (per D-06).

    Per D-06: this test confirms the pre-capture contract at the Rust level. No change to production Rust code is required.
