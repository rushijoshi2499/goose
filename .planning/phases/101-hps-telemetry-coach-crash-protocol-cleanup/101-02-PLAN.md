  <action>
Read `GooseSwift/CoachChatModel.swift` in full before editing (273 lines — confirmed by research).

The root cause (per D-03/D-04 and RESEARCH.md Track 2): `startOAuthSignIn()` spawns a `Task` each time it is called. Each task calls `chatGPT.startOAuthSignIn()` which internally polls the Codex OAuth endpoint via `authClient.completeDeviceCodeLogin(code)`. There is no timeout and no cancellation of the previous task, so repeated taps stack polling loops indefinitely, freezing the UI at "Waiting for approval."

**Fix — two changes only:**

1. **Add a new stored property** alongside the existing `sendTask` field (find `private var sendTask` and add immediately after it):
   ```
   private var signInTask: Task<Void, Never>?
   ```

2. **In `startOAuthSignIn()`**, before `signInTask = Task { ... }` (the new assignment), insert:
   ```
   signInTask?.cancel()
   ```
   Then assign the new task to `signInTask` instead of using an anonymous `Task { }`. The complete replacement for the existing anonymous `Task { [chatGPT, weak self] in ... }` block:
   ```
   signInTask = Task { [chatGPT, weak self] in
     do {
       try await chatGPT.startOAuthSignIn()
       self?.seedAssistantPromptIfNeeded()
     } catch is CancellationError {
       // cancelled — no-op
     } catch {
       self?.errorMessage = error.localizedDescription
     }
   }
   ```

Do NOT change the signature of `startOAuthSignIn()`, do NOT add `async` or `throws`, do NOT modify `ChatGPTCoachProvider`, do NOT add a timeout wrapper — the task cancellation pattern is sufficient (per D-04: fix must eliminate the crash path; the stacked-task freeze is the crash path).

The `catch is CancellationError` arm is already present in the existing anonymous task — preserve it. Error handling for non-cancellation errors also unchanged.
