# Plan 111-01: Protocol Offset WHY Comments

## Objective

Add inline `// offset N:` WHY comments at every empirical WHOOP byte-offset read site in `protocol.rs` and `bridge/mod.rs`. Comments explain *why* an offset is that value (empirical derivation, hardware-verified layout) — not *what* the code does. Zero logic changes. `cargo test --locked` passes unchanged.

## Context

Phase 86 and 108 already added doc-comment blocks (`///`) above several parser functions. What remains is adding inline `// offset N:` anchors *at the actual array read sites* inside function bodies. The doc block above the function is context; the inline comment at the read is the anchor that helps a reader scanning inside a long function.

**Format (D-01 from CONTEXT.md):**
```
// offset N: <type>, <field>; <why-non-obvious>
//   <empirical derivation note>; verified via hardware captures YYYY-MM-DD
```

**Rules (D-02/D-03):**
- WHY only — no WHAT
- No RE tool names — use "hardware captures", "protocol observation", "BLE capture analysis"
- Neutral language throughout
