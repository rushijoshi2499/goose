# Milestone v9.0: BLE Reliability & Protocol Parity

**Status:** ✅ SHIPPED 2026-06-11
**Phases:** 61–65 (Phase 66 deferred — hardware gate)
**Total Plans:** 10

## Overview

Closed the critical architectural gaps identified by Ghidra reverse engineering of WHOOP v5.37.0. Phases derived from `.planning/research/whoop-re/WHOOP-GOOSE-CROSS-COMPARE.md`. Introduced formal BLE bonding state machine (WHPBLEBondingManager parity), per-sensor upload watermarks (WHPStrapLatestUploadedMetricDateKey parity), NWPathMonitor-based upload gating with exponential backoff (WHPNetworkMonitor parity), Swift-side HR spike filter (WHPHeartRateDataSanitizer parity), and a generic BLE state machine type (WHPStateMachine parity). Phase 60 (Band-First Sync) was shipped with v8.0 and is counted as the transition phase.

### Phase 61: BLE Bonding State Machine

**Goal:** Replace implicit OS bonding path with a formal 5-state GooseBLEBondingManager (WHPBLEBondingManager parity).
**Depends on:** Phase 60
**Plans:** 3/3 complete

Plans:
- [x] 61-01: Foundation types — GooseBLEBondingState enum + GooseBLEBondingManager + LocalizedStatusStrings

- [x] 61-03: Human-verify checkpoint — bond-loss auto-recovery and persistence confirmed

### Phase 62: Upload Watermark per Sensor

**Goal:** Track last successfully uploaded timestamp per data type to prevent re-uploads after crash/restart.
**Depends on:** Phase 61
**Plans:** 2/2 complete

Plans:
- [x] 62-01: GooseUploadWatermark store (enum WatermarkType + UserDefaults persistence + clearAllWatermarks)
- [x] 62-02: Wire into upload pipeline (effectiveSince gate, write on 2xx, clearAllUploadWatermarks on GooseAppModel)

**Key decisions:**
- Two independent watermarks: rawFrames + decodedStreams (fail independently)
- Watermark value = max(row.ts) of uploaded rows, not Date() (prevents backfill gaps)
- since_ts passed to Rust rows_pending_upload to prevent limit=500 filling with old rows
- 4xx classified as clientError — no retry (only 5xx retried)

### Phase 63: Network Monitor & Upload Gating

**Goal:** Gate uploads on NWPathMonitor reachability; exponential backoff for 5xx failures.
**Depends on:** Phase 62
**Plans:** 2/2 complete

Plans:
- [x] 63-01: GooseNetworkMonitor — NWPathMonitor wrapper with NSLock + GooseAppModel.isNetworkReachable
- [x] 63-02: Wire reachability gate + APNs token gate + exponential backoff + pendingAPNSToken race fix

**Key decisions:**

## Milestone Summary

**Key Decisions:**
- NSLock used throughout new BLE subsystem types (consistent with GooseNetworkMonitor, GooseBLEBondingManager)
- Generic StateMachine<State, Event> struct — minimal scope (BLE bonding only, per ROADMAP note)
- Upload watermarks: per-type (rawFrames + decodedStreams), not per-device (sufficient for single-WHOOP use case)
- All (20...240) HR literals replaced with GooseHRSanitizer thresholds

**Issues Resolved:**
- BLE bond loss caused silent reconnect loops → GooseBLEBondingManager with isBondLossError (CBError 14/15 + encryptionTimedOut)
- Upload re-sending all data after crash → watermark gate prevents duplicate TimescaleDB rows
- Uploads attempted offline → NWPathMonitor gate + hasPendingUploadAfterReconnect retry
- HR spikes reaching algorithms → GooseHRSanitizer single-chokepoint filter

**Issues Deferred:**
- CAPSENSE-01 (Phase 66) — hardware gate, UUID not identifiable statically
- clearAllUploadWatermarks() has no UI call site — needs future logout/device-swap UI
- DispatchSemaphore in runHealthCheck — pre-existing, not v9.0 scope
- Per-device watermarks — stretch goal for multi-WHOOP scenarios

**Technical Debt Incurred:**
- pendingAPNSToken static cache in GooseAppDelegate — functional but fragile for sharedModel init race
- static var bleLogger (computed) in StateMachine<> — Swift limitation, not ideal but correct
- `isReachable = true` on GooseNetworkMonitor init — trusts network until first path update

**Build state:** BUILD SUCCEEDED on iPhone 17 simulator (iOS 26.0) throughout all phases.
**Xcconfig signing:** Config/Local.xcconfig (gitignored) pattern added for per-developer bundle ID override.
