<objective>
Fix SYNC-07: Gen4 historical sync on service UUID 61080005 (WHOOP_DATA_FROM_STRAP_GEN4) drops
packet47 body rows. Research identified two candidate root causes:

  Candidate A (body_hex suppression): protocol.rs line 696 suppresses body_hex for packet_k in
  {10, 21, 24}. The metric extractor reads from payload_hex (not body_hex), so suppression does
  NOT break extraction — but the suppression may cause the store to record a row with an empty
  body_hex, making the body unreadable for any future tooling that reads body_hex.

  Candidate B (frame routing): Gen4 historical frames may not reach import_captured_frame_timed
  at all — due to Swift routing (isHistoricalSyncing guard, UUID subscription gap) or Rust-side
  device_type mismatch.

This plan's first task writes a targeted integration test that calls import_captured_frame_timed
directly with a synthetic Gen4 packet47 frame — confirming whether the Rust import path produces
a decoded_frames row. The test result determines which candidate is the true root cause and
which code edit is needed.

Purpose: Give Gen4 users complete historical sync data. packet47 body rows are required for
overnight sleep metric extraction.

Output: Integration test confirming Gen4 historical import; targeted fix in protocol.rs or
a documented routing issue requiring a Swift fix in a follow-up phase.
</objective>

    Building the synthetic Gen4 frame:
    The Gen4 frame format (from protocol.rs lines 419–443):
      [0xAA][len_lo][len_hi][CRC8_of_header][payload bytes...][CRC32 x4 bytes]
    The payload for a historical data packet:
      payload[0] = 0x2F  (packet_type = historicalData = 47)
      payload[1] = 24    (packet_k = V24History)
      payload[2] = 0     (flags/status)
      payload[3..] = V24 body bytes (at minimum 77 bytes for a valid V24 body)
    Total minimum payload length: 80 bytes (3 header + 77 body).

    To compute CRC8: look for a helper function used in existing Gen4 tests or in protocol.rs
    test helpers. If no helper is pub, set the CRC8 byte to 0 and check if parse_frame
    has a test mode that skips CRC validation — if so, use that. If CRC validation is always
    enforced, look for a crc8 calculation in the codebase and replicate it.

    For CRC32: same approach — find the existing crc32 helper or use the crc32fast crate
    (already in Cargo.toml) to compute the trailer bytes over the full frame up to the trailer.

    The test setup:
    1. Open a GooseStore on a tempfile DB (use tempfile::NamedTempFile as existing tests do).
    2. Insert a CaptureSessionInput to give the frame a session to belong to.
    3. Call import_captured_frame_timed (or equivalent bridge function) with:
       - frame_hex: hex::encode(synthetic_gen4_frame)
       - service_uuid: "61080005-8d6d-82b8-614a-1c8cb0f8dcc6"
       - characteristic_uuid: appropriate characteristic UUID for Gen4 data
       - device_type: "GEN4"
    4. Assert result is Ok.
    5. Query decoded_frames table: SELECT COUNT(*) WHERE service_uuid LIKE '61080005%' AND
       packet_type = 47. Assert count >= 1.

    For packet47_count_gen4: after importing, query sleep_sessions for the packet47_count field
    and assert it is >= 1. Look at existing store_tests.rs patterns for how sleep sessions are
    queried.

    If import_captured_frame_timed is not directly callable from integration tests (private or
    requires bridge setup), use the bridge JSON method "capture.import_frame_batch" through
    GooseRustBridge pattern (look at v24_biometric_bridge_tests.rs for how bridge methods are
    called in tests).

    Body_hex suppression fix (apply ONLY if Test A fails specifically because body_hex="" causes
    insert_decoded_frame to skip the row):
    In Rust/core/src/protocol.rs at the body_hex suppression block (line 696):
    Change the suppression to apply only when a flag indicates live/streaming context.
    The approach: examine whether parse_frame receives a context parameter that distinguishes
    historical from live. If not, check whether the caller (import_captured_frame_timed) can
